1 · Spring Grundlagen

Spring ist ein Open-Source-Framework zur plattformunabhängigen Entwicklung von Web-Applikationen mit Java. Es basiert auf vier Kernprinzipien:

Convention over Configuration
Best-Practice-Annahmen sind voreingestellt; Entwickler spezifizieren nur Abweichungen vom Standard. Weniger Konfigurationsaufwand.
Modularität
Spring ist in Module aufgeteilt, die nur bei Bedarf geladen werden (Web, Data, Security …).
Schichtenarchitektur
Klare Trennung von Presentation, Business Logic und Data Storage.
Dependency Injection
Spring übernimmt die Verwaltung von Objekten, um Komplexität zu reduzieren.
Spring Boot Baut auf dem Spring Framework auf und nutzt Standardkonfigurationen, um selbstständig ausführbare Web-Applikationen als JAR-Datei schnell zu entwickeln. Früher war ein dedizierter Applikationsserver (z. B. Tomcat) nötig – Spring Boot bringt ihn mit.

Schichtenarchitektur

Spring Boot strukturiert Anwendungen in vier funktionale Schichten (Layer):

Layer Annotation Aufgabe
Controller @Controller / @RestController Verarbeitet Client-Anfragen (HTTP-Requests) und gibt Antworten zurück (HTML, JSON, …)
Service @Service Kapselt und implementiert die Geschäftslogik
Repository @Repository Verwaltet Datenzugriff und -speicherung (Datenbankoperationen)
Entity / Model @Entity Repräsentiert Datenbankobjekte (Tabellen)
Älteres 3-Schichten-Modell (Grundprinzip)
  • Presentation Layer – Anzeige & Eingabe (Frontend)
  • Business Logic Layer – Geschäftslogik
  • Data Storage Layer – Datenspeicherung & Datenbankzugriff

Spring Beans

Was ist ein Bean?
Ein Bean ist eine Java-Klasse, deren Lebenszyklus (Erstellung, Konfiguration, Skalierung, Löschung) vollständig vom Spring-Framework verwaltet wird.

Beans werden nicht manuell per new instanziiert – Spring übernimmt das im sogenannten IoC-Container (Application Context).

Beans werden wie normale Java-Klassen geschrieben und über Stereotyp-Annotationen als Bean deklariert:

@Service
public class MovieServiceImpl implements MovieService {
    // Spring erstellt und verwaltet diese Klasse automatisch
}

Stereotype-Annotationen (Bean-Typen)

@Component ist die Basis-Annotation. Alle anderen Stereotypen sind spezialisierte Untertypen:

@Component – generischer Spring-Bean (keine Schichtzuordnung)
├── @Controller – verarbeitet Client-Anfragen / gibt Antworten zurück
│   └── @RestController – wie @Controller, Ausgabe als JSON
├── @Service – implementiert Geschäftslogik
├── @Repository – Datenbankzugriff & -verwaltung
└── @Configuration – Infrastruktur-Konfiguration ("Configuration as Code")
Annotation Schicht Zweck & Details
@Component Generisch Basis für alle Stereotypen. Markiert eine Klasse als Spring-verwalteten Bean. Kein expliziter Layer.
@Controller Presentation Kommuniziert mit dem Benutzer/API. Nimmt HTTP-Requests entgegen und gibt Antworten in verschiedenen Formaten zurück (HTML, JSON, …).
@RestController Presentation Unterklasse von @Controller. Gibt automatisch JSON zurück (kombiniert @Controller + @ResponseBody).
@Service Business Logic Kapselt Geschäftslogik. Wird von Controllern aufgerufen. Kein direkter Datenbankzugriff.
@Repository Data Access Datenspeicherung und -verwaltung. Kapselt Datenbankzugriffe. Spring übersetzt DB-Exceptions in Spring-Exceptions.
@Configuration Infrastruktur "Configuration as Code"-Paradigma. Konfiguriert Spring-Infrastruktur programmatisch.

Dependency Injection (DI)

Was ist Dependency Injection?
DI ist eine Implementierung des IoC-Prinzips. Spring übernimmt zur Laufzeit die Instanziierung und Zuweisung von Abhängigkeiten. Eine Klasse erhält die @Autowired-Annotation → Spring findet automatisch eine passende Implementierung und injiziert sie.

Es gibt drei Arten der Dependency Injection:

1. Constructor Injection ✓
Best Practice seit Spring 3.5
Abhängigkeiten werden über den Konstruktor übergeben. 
@Service
public class MovieControllerImpl {
  private final MovieService svc;

  // @Autowired optional ab Spring 3.5
  public MovieControllerImpl(
      MovieService svc) {
    this.svc = svc;
  }
}
2. Setter Injection
Abhängigkeiten werden nach der Instanziierung über Setter-Methoden gesetzt. 
@Service
public class MyService {
  private Dep dep;

  @Autowired
  public void setDep(Dep dep) {
    this.dep = dep;
  }
}
3. Field Injection ✗
Direkt auf das Feld annotiert. Nicht empfohlen (schlechte Testbarkeit, versteckte Abhängigkeiten). 
@Service
public class MyService {
  @Autowired
  private Dep dep; // nicht empfohlen
}

Inversion of Control (IoC)

IoC-Prinzip
Kernparadigma: Objekte sollen so lose gekoppelt (entkoppelt) wie möglich sein.
Grundidee: Klassen nutzen andere Klassen, ohne deren Implementierungsdetails kennen zu müssen.

Ohne IoC: Klasse Car kennt und instanziiert PetrolEngine direkt → starke Kopplung.

Mit IoC (Interface): Car referenziert nur das Interface Engine → beliebige Implementierung austauschbar.

// Ohne IoC – stark gekoppelt (SCHLECHT)
public class Car {
    private PetrolEngine engine = new PetrolEngine(); // direkte Abhängigkeit
}

// Mit IoC + DI – lose gekoppelt (GUT)
public class Car {
    private Engine engine; // Interface, nicht konkrete Klasse

    public Car(Engine engine) { // Spring injiziert die Implementierung
        this.engine = engine;
    }
}
Vorteile von Dependency Injection:
  • Entkoppelte Weiterentwicklung interagierender Klassen
  • Reduzierte Code-Komplexität durch Schichtentrennung
  • Verbesserte Testbarkeit (lose Kopplung ermöglicht Test-Fassaden / Mocking)
  • Implementierungen zur Laufzeit austauschbar

Component Scan

Beim Anwendungsstart führt Spring einen Component Scan durch:
  1. Spring scannt alle Klassen im Classpath nach @Component und Untertypen
  2. Gefundene Klassen werden im Application Context (IoC-Container) als Beans registriert
  3. Abhängigkeiten zwischen Beans werden erkannt und automatisch injiziert
Wichtig: Das Paket mit @SpringBootApplication (inkl. Unterpakete) wird automatisch gescannt. Klassen außerhalb dieses Pakets werden NICHT gefunden.

AnnotationZielBedeutung
@SpringBootApplicationKlasseKombiniert @Configuration + @EnableAutoConfiguration + @ComponentScan. Einstiegspunkt der Anwendung.
@ComponentKlasseGenerischer Spring-Bean. Lebenszyklus von Spring verwaltet.
@ControllerKlassePresentation-Layer. Verarbeitet HTTP-Requests.
@RestControllerKlasseWie @Controller, antwortet automatisch mit JSON.
@ServiceKlasseBusiness-Logic-Layer.
@RepositoryKlasse/InterfaceData-Access-Layer. Datenbankzugriff.
@ConfigurationKlasseKonfigurationsklasse (Configuration as Code).

2 · ORM, JPA & Spring Data

Das Problem: OOP vs. Relationale DB

Objektorientierte Anwendung (OOP)
  • Entitäten als Objekte modelliert
  • Beziehungen über Referenzen/Relationen
  • Klassen mit Verhalten (Methoden)
Relationale Datenbank (SQL)
  • Tabellen im relationalen Schema
  • Beziehungen über Foreign Keys
  • Fokus auf Normalformen & Speichereffizienz

→ Diese grundlegend verschiedenen Paradigmen überbrückt das Object-Relational Mapping (ORM).

ORM (Object-Relational Mapping)

ORM übersetzt zwischen Java-Objekten und relationalen Tabellen:
  • Java-Klassen → Datenbank-Tabellen
  • Klassen-Attribute → Tabellen-Spalten
  • Objekt-Beziehungen → Foreign Keys

ORM-Operationen

OperationBedeutung
createAnwendungsobjekt wird in DB eingefügt (INSERT)
readDB-Datensatz wird zu Anwendungsobjekt konvertiert (SELECT)
updateÄnderungen am Objekt werden in DB synchronisiert (UPDATE)
deleteAnwendungsobjekt wird aus DB gelöscht (DELETE)
Vorteile ORM
  • Harmonisierte DB-Struktur
  • Hoher Automatisierungsgrad
  • Reduzierte Datenkomplexität
  • Schutz vor SQL-Injection
  • Einfacher DB-Wechsel
Nachteile ORM
  • DB-Abstraktion kann Performance kosten
  • Initiale Konfiguration komplex
  • SQL-Kenntnisse weiterhin nötig

Glossar: JPA-Technologiestack

BegriffErklärung
JPA (Java Persistence API)Spezifikation/Standard für die Beschreibung von Entities und ORM in Java. Definiert Annotationen, Interfaces und Verhalten – keine Implementierung.
Spring Data JPAImplementiert den JPA-Standard und bietet zusätzliche Komfortfunktionen (Repositories, Derived Queries …).
HibernatePopuläres ORM-Framework; die Implementierung hinter Spring Data JPA.
JDBCLow-Level-Framework für direkten Datenbankzugriff (SQL-Statements).
JDBC-DriverVerbindet JDBC mit der konkreten Datenbank (H2, MySQL, PostgreSQL …).

Entities

Was ist eine Entity?
Eine Entity ist ein Datenobjekt, das in der Datenbank gespeichert wird. Sie wird mit @Entity markiert und im Paket model abgelegt.

Pflichtanforderungen einer Entity (JPA-Spezifikation):

  1. Leerer Konstruktor (No-Arg Constructor) – Pflicht laut JPA-Spezifikation
  2. Getter & Setter für alle Attribute (manuell oder per Lombok)
  3. Identifikationsschlüssel mit @Id und @GeneratedValue

Entity-Annotationen im Überblick

AnnotationBedeutung
@EntityMarkiert die Klasse als persistierbare Datenbankentität
@IdIdentifiziert das Attribut als Primärschlüssel (Identifikationsschlüssel)
@GeneratedValueSchlüssel wird von Hibernate auto-generiert (verhindert Kollisionen)
@ColumnMappt Attribut auf Tabellenspalte (optional, für Umbenennung/Constraints)
@TableDefiniert Tabellenname (optional)
@JoinColumnDefiniert die Foreign-Key-Spalte für eine Beziehung
@JoinTableDefiniert eine Zwischentabelle (für @ManyToMany)
// Vollständiges Entity-Beispiel aus dem Projekt
@Entity
@Getter @Setter @NoArgsConstructor  // Lombok
public class Movie {

    @Id
    @GeneratedValue
    private long id;           // Primärschlüssel (auto-generiert)

    private String title;
    private double duration;   // in Stunden
    private LocalDate premiereDate;
    private String description;
    private String trailerUrl;
    private String coverImgUrl;
    private double imdbRating;
}

Repositories

Was ist ein Repository?
Ein Repository abstrahiert den Datenbankzugriff für eine Entity. Es enthält alle Funktionen zum Speichern, Abrufen und Löschen von Entities.

No-Code-Ansatz: Nur das Interface muss definiert werden – Spring generiert die Implementierung automatisch.

Ein Repository erweitert JpaRepository<EntityTyp, PrimärschlüsselTyp>:

@Repository
public interface MovieRepository extends JpaRepository<Movie, Long> {
    // Spring implementiert automatisch: save, findById, findAll, delete, ...
}

delete(entity)DELETEEntity löschen

count()SELECT COUNT(*)Anzahl der Einträge

JPQL & @Query-Annotation

Was ist JPQL?
JPQL (Java Persistence Query Language) ist eine SQL-ähnliche Sprache, die aber auf dem Objektgraphen operiert – d. h. Klassenamen und Attribute statt Tabellennamen und Spalten.
Wichtig: JPQL-Queries werden beim Spring Boot Start validiert. Syntaxfehler fallen sofort beim Start auf.
AnnotationBedeutung
@Query("...")Definiert eine manuelle JPQL-Abfrage für eine Repository-Methode
@Param("name")Bindet einen Methodenparameter an einen benannten JPQL-Parameter (:name)
// Einfache JPQL-Abfrage (Aufgabe 2)
@Query("SELECT m FROM Movie m WHERE m.title = :param1")
List<Movie> findAllMoviesByTitle(@Param("param1") String title);

// LIKE-Abfrage mit Wildcard
@Query("SELECT m FROM Movie m WHERE m.title like %:param1%")
List<Movie> findAllMoviesBelongingToSeries(@Param("param1") String series);

// JOIN über Kollektion (ManyToMany)
@Query("SELECT m FROM Movie m JOIN m.involvedMovieCast c WHERE c.surname = :param1")
List<Movie> findAllMoviesWithCastSurname(@Param("param1") String castSurname);

// Verschachtelter Attributzugriff (über Objekt-Referenz)
@Query("SELECT m FROM Movie m WHERE m.soundtrack.author_surname = :param1")
List<Movie> findAllMoviesWithSoundtrackAuthor(@Param("param1") String surname);

// Kollektionsgröße abfragen
@Query("SELECT m FROM Movie m WHERE size(m.involvedMovieCast) > :param1")
List<Movie> findAllMoviesWithMoreThanGivenCasts(@Param("param1") Integer num);

// JOIN über Award-Kollektion
@Query("SELECT m FROM Movie m JOIN m.awards a WHERE a.academy = :param1")
List<Movie> findAllMoviesWithAcademy(@Param("param1") String academy);

// Komplexe Abfrage: mehrere JOINs und Bedingungen
@Query("SELECT m FROM Movie m JOIN m.involvedMovieCast c JOIN m.awards a " +
       "WHERE c.surname = :param1 AND a.academy = :param2")
List<Movie> myMoviesByCastAndAward(@Param("param1") String surname,
                                    @Param("param2") String academy);

// Komplexe Abfrage: Award + Datum-Bereich
@Query("SELECT m FROM Movie m JOIN m.awards a WHERE " +
       "a.category = :param1 AND size(m.involvedMovieCast) > 0 AND " +
       "m.soundtrack.releaseDate > :param2 AND m.soundtrack.releaseDate < :param3")
List<Movie> findMoviesByCastSizeAndSoundtrack(@Param("param1") String category,
                                               @Param("param2") LocalDate minDate,
                                               @Param("param3") LocalDate maxDate);

3 · JPA-Beziehungstypen

Entities stehen in Beziehungen zueinander. Die Kardinalität gibt an, wie viele Entities einer Art mit Entities einer anderen Art in Beziehung stehen.

AnnotationKardinalitätBeispielDB-Umsetzung
@ManyToOneN:1Viele Movies → ein DirectorForeign Key in Movie-Tabelle
@OneToMany1:NEin Director → viele MoviesGegenseite zu @ManyToOne
@ManyToManyM:NViele Movies ↔ viele SchauspielerZwischentabelle (Join Table)
@OneToOne1:1Ein Movie ↔ ein SoundtrackForeign Key auf einer Seite

@ManyToOne / @OneToMany – 1:N-Beziehung

Beispiel: Ein Director hat viele Movies. Jeder Movie hat genau einen Director.
→ Der Movie ist der „Besitzer" der Beziehung (hat den Foreign Key director_id). 
// Movie-Entity (viele Movies → ein Director)  → "Many"-Seite
@ManyToOne(cascade = CascadeType.ALL)
@JoinColumn(name = "director_id", nullable = true)  // FK-Spalte in Movie-Tabelle
private Director director;

// Director-Entity (ein Director → viele Movies)  → "One"-Seite
@OneToMany(mappedBy = "director", cascade = CascadeType.ALL)
private List<Movie> movies;
// mappedBy = Name des Feldes auf der Besitzer-Seite
mappedBy: Gibt an, dass diese Seite nicht der Beziehungsbesitzer ist. Die angegebene Feld-Name ("director") zeigt, wo die Beziehung definiert (besessen) wird.

@ManyToMany – M:N-Beziehung

Beispiel: Viele Movies haben viele Schauspieler (MovieCast). Ein Schauspieler wirkt in mehreren Movies mit.
→ Datenbank benötigt eine Zwischentabelle (movie_moviecast). 
// Movie-Entity (Besitzer-Seite, definiert die Join-Tabelle)
@ManyToMany(cascade = CascadeType.ALL)
@JoinTable(
    name = "movie_moviecast",           // Name der Zwischentabelle
    joinColumns = {
        @JoinColumn(name = "movie_id", referencedColumnName = "id")   // FK für Movie
    },
    inverseJoinColumns = {
        @JoinColumn(name = "cast_id",  referencedColumnName = "id")   // FK für Cast
    }
)
private List<MovieCast> involvedMovieCast;

// MovieCast-Entity (nicht-besitzende Seite)
@ManyToMany(mappedBy = "involvedMovieCast", cascade = CascadeType.PERSIST)
private List<Movie> participatedMovies;
Wichtig @ManyToMany: Referenz muss immer von der Besitzer-Seite gesetzt werden (d. h. movie.getInvolvedMovieCast().add(castMember)). Nur dann wird die Zwischentabelle korrekt befüllt.

@OneToOne – 1:1-Beziehung

Beispiel: Jeder Movie hat genau einen Soundtrack. Jeder Soundtrack gehört zu genau einem Movie.
→ Soundtrack besitzt die Beziehung (hat den Foreign Key movie_id). 
// Soundtrack-Entity (Besitzer-Seite, hat den FK)
@OneToOne(cascade = CascadeType.ALL)
@JoinColumn(name = "movie_id")        // FK-Spalte in Soundtrack-Tabelle
private Movie movie;

// Movie-Entity (nicht-besitzende Seite)
@OneToOne(mappedBy = "movie", cascade = CascadeType.ALL)
private Soundtrack soundtrack;
Typische Falle: Bei @OneToOne müssen BEIDE Seiten gesetzt werden, bevor man persistiert: 
soundtrack.setMovie(movie);   // Besitzer-Seite setzen
movie.setSoundtrack(soundtrack); // Gegenseite setzen (für Bidirektionalität)
movieRepository.save(movie);     // Cascade speichert Soundtrack automatisch

Weitere Beziehung im Projekt: Movie ↔ Award (1:N)

// Movie-Entity
@OneToMany(mappedBy = "movie", cascade = CascadeType.ALL, fetch = FetchType.EAGER)
private List<Award> awards;

// Award-Entity (Besitzer-Seite)
@ManyToOne(cascade = CascadeType.ALL)
@JoinColumn(name = "movie_id")
private Movie movie;

Cascading (Kaskadierende Operationen)

Cascading definiert, wie sich Operationen an einer Entity auf referenzierte Entities auswirken.
CascadeTypeBedeutung
CascadeType.PERSISTWenn Owner gespeichert wird, wird auch die referenzierte Entity gespeichert
CascadeType.REMOVEWenn Owner gelöscht wird, wird auch die referenzierte Entity gelöscht
CascadeType.MERGEWenn Owner gemergt wird, wird auch die referenzierte Entity gemergt
CascadeType.REFRESHWenn Owner aktualisiert wird, wird auch die referenzierte Entity aktualisiert
CascadeType.DETACHWenn Owner vom Context getrennt wird, wird auch referenzierte Entity getrennt
CascadeType.ALLAlle obigen Operationen kaskadieren
Cascade-Strategie im Projekt:
CascadeType.ALL bei Movie→Director und Movie→MovieCast: Wenn Movie gespeichert wird, werden Director und Cast automatisch mitgespeichert.
CascadeType.PERSIST (nicht ALL) bei MovieCast→Movies: Cast wird gespeichert, aber Löschen von Cast löscht nicht automatisch Movies.

Lazy Loading vs. Eager Loading

Eager Loading
Alle referenzierten Objekte werden sofort beim Laden der Entity aus der DB geladen (eine große Abfrage / mehrere Abfragen).

Problem: Kann bei bidirektionalen Beziehungen zu enormen Performance-Problemen führen.

Default bei: @ManyToOne, @OneToOne
Lazy Loading ✓ (empfohlen)
Referenzierte Objekte werden erst beim Zugriff auf das Attribut geladen (Platzhalter zuerst).

Empfohlen für Collections.

Default bei: @OneToMany, @ManyToMany

Achtung: Lazy-Loading außerhalb eines Transaktionskontexts → LazyInitializationException! 
// Explizit Eager-Loading setzen
@OneToMany(mappedBy = "movie", cascade = CascadeType.ALL, fetch = FetchType.EAGER)
private List<Award> awards;

// Explizit Lazy-Loading (Standard, kann weggelassen werden)
@ManyToMany(cascade = CascadeType.PERSIST, fetch = FetchType.LAZY)
private List<Movie> participatedMovies;
Lazy Loading außerhalb einer Transaktion: Wenn eine lazy-geladene Collection außerhalb des Hibernate-Sessions (Persistenzkontext) zugegriffen wird, gibt es eine LazyInitializationException. Lösung: @Transactional auf die Service-Methode setzen!

Bidirektionale Beziehungen

Bei bidirektionalen Beziehungen müssen beide Seiten gepflegt werden:
  1. Die Besitzer-Seite (ohne mappedBy) trägt den Foreign Key und steuert die DB-Persistenz
  2. Beim Setzen der Referenz im Code muss immer die Besitzer-Seite gesetzt werden
  3. Für korrekte bidirektionale Navigation im Code sollte auch die andere Seite gesetzt werden
// Korrekte bidirektionale Beziehungspflege
Director director = new Director("Christopher", "Nolan");
Movie movie = new Movie("Interstellar");

movie.setDirector(director);           // Besitzer-Seite setzen (pflicht für DB)
director.getMovies().add(movie);       // Gegenseite setzen (für Java-Navigation)

movieRepository.save(movie);           // Director wird durch CascadeType.ALL mitgespeichert
// NICHT direktory separately speichern nötig!

4 · Transaktionen

Was ist eine Transaktion?
Eine Transaktion fasst zusammengehörige Operationen zu einer Einheit zusammen. Sie implementiert das Alles-oder-Nichts-Prinzip (All-or-Nothing): Entweder alle Operationen gelingen, oder bei einem Fehler wird alles rückgängig gemacht (Rollback).

ACID-Eigenschaften

A – Atomicity (Atomarität)
Eine Transaktion ist nur erfolgreich, wenn alle Operationen erfolgreich sind.
Bei Fehler → Rollback zum Ausgangszustand.
C – Consistency (Konsistenz)
Die Transaktion beginnt und endet in einem konsistenten DB-Zustand.
Alle Constraints/Bedingungen bleiben erhalten.
I – Isolation
Transaktion läuft unabhängig von parallel laufenden Transaktionen.
Keine Seiteneffekte durch parallele Transaktionen.
D – Durability (Dauerhaftigkeit)
Ergebnisse einer Transaktion sind permanent gespeichert und überleben Server-Abstürze.

@Transactional Annotation

@Transactional markiert eine Methode oder Klasse als transaktional. Spring öffnet automatisch einen Transaktionskontext vor der Methode und committed/rollt zurück nach der Methode.
AnwendungsortWirkung
Auf MethodeNur diese Methode läuft transaktional (empfohlen)
Auf KlasseAlle Methoden der Klasse laufen transaktional

Wichtiger Anwendungsfall: Lazy-Loading in Transaktionen

Lazy-geladene Collections können nur innerhalb eines aktiven Persistenzkontexts (Session) aufgelöst werden. @Transactional hält die Session offen:

@Service
public class MovieServiceImpl implements MovieService {

    private MovieRepository movieRepository;

    // Constructor Injection (Best Practice)
    public MovieServiceImpl(MovieRepository movieRepository) {
        this.movieRepository = movieRepository;
    }

    @Transactional  // Hält Persistenzkontext offen → Lazy-Loading möglich
    public List<String> getCastOfMovie(String movieTitle) {
        List<String> result = new LinkedList<>();
        List<Movie> tmp = this.movieRepository.findByTitleContaining(movieTitle);
        if (tmp.size() == 1) {
            // involvedMovieCast ist lazy → würde ohne @Transactional Exception werfen
            tmp.get(0).getInvolvedMovieCast().forEach(cast ->
                result.add(cast.getForename())
            );
        }
        return result;
    }

    @Transactional  // Automatisches Update: kein explizites save() nötig!
    public void changeMovieTitle(long id, String newMovieName) {
        if (!newMovieName.isEmpty()) {
            Movie tmp = this.movieRepository.findById(id).get();
            tmp.setTitle(newMovieName);
            // Das Objekt ist managed → Hibernate erkennt Änderung automatisch
            // KEIN movieRepository.save(tmp) nötig!
        }
    }
}
Dirty Checking: Innerhalb einer Transaktion werden Änderungen an einem managed Entity automatisch beim Commit persistiert – ohne explizites save().
Standard-Transaktionen: Alle CRUD-Operationen des JpaRepository (save, findById, findAll, delete) führen automatisch als Transaktionen aus.

@Getter @Setter @NoArgsConstructor
public class MovieCast {
    @Id @GeneratedValue
    private long id;
    private String forename;
    private String surname;

    @ManyToMany(mappedBy = "involvedMovieCast", cascade = CascadeType.PERSIST)
    private List<Movie> participatedMovies;

@Entity
@Getter @Setter @NoArgsConstructor
public class Soundtrack {
    @Id @GeneratedValue
    private long id;
    private String name;
    private String author_forename;
    private String author_surname;
    private LocalDate releaseDate;

    @OneToOne(cascade = CascadeType.ALL)
    @JoinColumn(name = "movie_id")   // FK in Soundtrack-Tabelle
    private Movie movie;
}

// Award Entity (nur in Lösung)
@Entity
@Getter @Setter @NoArgsConstructor
public class Award {
    @Id @GeneratedValue
    private long id;
    private String academy;
    private String category;
    private int yearAwarded;

    @ManyToOne(cascade = CascadeType.ALL)
    @JoinColumn(name = "movie_id")
    private Movie movie;
}

// Movie Entity – vollständig mit allen Beziehungen
@Entity
@Getter @Setter @NoArgsConstructor
public class Movie {
    @Id @GeneratedValue
    private long id;
    private String title;
    private double duration;
    private LocalDate premiereDate;
    private String description;
    private String trailerUrl;
    private String coverImgUrl;
    private double imdbRating;

    // N:1 zu Director
    @ManyToOne(cascade = CascadeType.ALL)
    @JoinColumn(name = "director_id", nullable = true)
    private Director director;

    // M:N zu MovieCast (Besitzer-Seite)
    @ManyToMany(cascade = CascadeType.ALL)
    @JoinTable(
        name = "movie_moviecast",
        joinColumns =        { @JoinColumn(name = "movie_id",  referencedColumnName = "id") },
        inverseJoinColumns = { @JoinColumn(name = "cast_id",   referencedColumnName = "id") }
    )
    private List<MovieCast> involvedMovieCast;

    // 1:1 zu Soundtrack (nicht-besitzende Seite)
    @OneToOne(mappedBy = "movie", cascade = CascadeType.ALL)
    private Soundtrack soundtrack;

    // 1:N zu Award (nicht-besitzende Seite)
    @OneToMany(mappedBy = "movie", cascade = CascadeType.ALL, fetch = FetchType.EAGER)
    private List<Award> awards;
}

@Component

                           MovieService movieService) {

        Director nolan = new Director(); nolan.setForename("Christopher"); nolan.setSurname("Nolan");
        Director wachowski = new Director(); wachowski.setForename("Lana"); wachowski.setSurname("Wachowski");

        // Schauspieler anlegen
        MovieCast actor1 = new MovieCast(); actor1.setForename("Matthew"); actor1.setSurname("McConaughey");
        MovieCast actor2 = new MovieCast(); actor2.setForename("Christian");  actor2.setSurname("Bale");

        // Soundtracks anlegen
        Soundtrack st1 = new Soundtrack(); st1.setName("Interstellar OST");
        st1.setAuthor_forename("Hans"); st1.setAuthor_surname("Zimmer");

        // Filme anlegen
        Movie interstellar = new Movie(); interstellar.setTitle("Interstellar"); interstellar.setImdbRating(8.7);
        Movie batman = new Movie(); batman.setTitle("The Dark Knight"); batman.setImdbRating(9.0);

        // Beziehungen setzen

        movieRepository.save(interstellar);

        log.info("Alle Filme: {}", movieRepository.findAll().stream().map(Movie::getTitle).toList());
