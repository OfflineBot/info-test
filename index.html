1 · Spring Grundlagen

Spring ist ein riesiges Werkzeugkasten für Java-Entwickler. Statt alles selbst zu verdrahten (wer ruft wen auf? wer erstellt welches Objekt?), übernimmt Spring das für dich. Du sagst Spring was du brauchst, Spring kümmert sich um das Wie.

Spring ist ein Open-Source-Framework zur plattformunabhängigen Entwicklung von Web-Applikationen mit Java. Es basiert auf vier Kernprinzipien:

Convention over Configuration
Best-Practice-Annahmen sind voreingestellt; Entwickler spezifizieren nur Abweichungen vom Standard. Weniger Konfigurationsaufwand.
Modularität
Spring ist in Module aufgeteilt, die nur bei Bedarf geladen werden (Web, Data, Security …).
Schichtenarchitektur
Klare Trennung von Presentation, Business Logic und Data Storage.
Dependency Injection
Spring übernimmt die Verwaltung von Objekten, um Komplexität zu reduzieren.
Spring Boot Baut auf dem Spring Framework auf und nutzt Standardkonfigurationen, um selbstständig ausführbare Web-Applikationen als JAR-Datei schnell zu entwickeln. Früher war ein dedizierter Applikationsserver (z. B. Tomcat) nötig – Spring Boot bringt ihn bereits mit.

Spring Boot ist Spring mit sinnvollen Voreinstellungen. Du startest eine fertige App mit einem einzigen java -jar app.jar – kein extra Tomcat installieren nötig.

Schichtenarchitektur

Die App ist in Schichten aufgeteilt, damit jede Klasse genau eine Aufgabe hat. Der Controller redet mit dem Browser, der Service denkt nach, das Repository spricht mit der Datenbank – und keiner mischt sich in die Arbeit des anderen ein.

Spring Boot strukturiert Anwendungen in vier funktionale Schichten (Layer):

LayerAnnotationAufgabe
Controller @Controller / @RestController Verarbeitet Client-Anfragen (HTTP-Requests) und gibt Antworten zurück (HTML, JSON, …)
Service @Service Kapselt und implementiert die Geschäftslogik
Repository @Repository Verwaltet Datenzugriff und -speicherung (Datenbankoperationen)
Entity / Model @Entity Repräsentiert Datenbankobjekte (Tabellen)
Älteres 3-Schichten-Modell (Grundprinzip)
  • Presentation Layer – Anzeige & Eingabe (Frontend)
  • Business Logic Layer – Geschäftslogik
  • Data Storage Layer – Datenspeicherung & Datenbankzugriff

Spring Beans

Normalerweise erstellst du Objekte mit new MeineKlasse(). Bei Spring machst du das nicht mehr selbst – du markierst die Klasse mit einer Annotation und Spring erstellt und verwaltet das Objekt für dich. Dieses von Spring verwaltete Objekt heißt Bean.

Vergleich: Du gehst in ein Restaurant. Du bestellst (=Annotation) und der Kellner (=Spring) bringt dir das Essen (=Objekt). Du kochst nicht selbst.

Was ist ein Bean?
Ein Bean ist eine Java-Klasse, deren Lebenszyklus (Erstellung, Konfiguration, Skalierung, Löschung) vollständig vom Spring-Framework verwaltet wird.

Beans werden nicht manuell per new instanziiert – Spring übernimmt das im sogenannten IoC-Container (Application Context).

Beans werden wie normale Java-Klassen geschrieben und über Stereotyp-Annotationen als Bean deklariert:

@Service
public class MovieServiceImpl implements MovieService {
    // Spring erstellt und verwaltet diese Klasse automatisch
    // Kein "new MovieServiceImpl()" irgendwo nötig!
}

Stereotype-Annotationen (Bean-Typen)

Stereotypen sind wie Berufsbezeichnungen für deine Klassen. @Service bedeutet "ich bin ein Buchhalter (Geschäftslogik)", @Repository bedeutet "ich bin ein Archivar (Datenbank)", @Controller bedeutet "ich bin die Rezeption (HTTP-Anfragen)".

Alle sind letztendlich @Component – das ist der Oberbegriff für "von Spring verwaltetes Objekt".

@Component ist die Basis-Annotation. Alle anderen Stereotypen sind spezialisierte Untertypen:

@Component – generischer Spring-Bean (keine Schichtzuordnung)
├── @Controller – verarbeitet Client-Anfragen / gibt Antworten zurück
│   └── @RestController – wie @Controller, Ausgabe als JSON
├── @Service – implementiert Geschäftslogik
├── @Repository – Datenbankzugriff & -verwaltung
└── @Configuration – Infrastruktur-Konfiguration ("Configuration as Code")
AnnotationSchichtZweck & Details
@ComponentGenerisch Basis für alle Stereotypen. Markiert eine Klasse als Spring-verwalteten Bean. Kein expliziter Layer. Verwenden wenn keine spezialisierte Annotation passt.
@ControllerPresentation Kommuniziert mit dem Benutzer/API. Nimmt HTTP-Requests entgegen und gibt Antworten in verschiedenen Formaten zurück (HTML, JSON, …).
@RestControllerPresentation Unterklasse von @Controller. Gibt automatisch JSON zurück (kombiniert @Controller + @ResponseBody). Für REST-APIs.
@ServiceBusiness Logic Kapselt Geschäftslogik. Wird von Controllern aufgerufen. Kein direkter Datenbankzugriff – das macht das Repository.
@RepositoryData Access Datenspeicherung und -verwaltung. Kapselt Datenbankzugriffe. Spring übersetzt DB-Exceptions automatisch in Spring-Exceptions.
@ConfigurationInfrastruktur "Configuration as Code"-Paradigma. Konfiguriert Spring-Infrastruktur programmatisch statt in XML-Dateien.

Dependency Injection (DI)

Stell dir vor, ein Auto baut sich seinen Motor selbst – das wäre schlechtes Design. Besser: Der Motor wird dem Auto von außen gegeben. Das ist Dependency Injection.

In Java bedeutet das: Eine Klasse bekommt ihre Abhängigkeiten (andere Objekte die sie braucht) von Spring automatisch "injiziert" (hineingegeben) – sie baut sie nicht selbst mit new.

Was ist Dependency Injection?
DI ist eine Implementierung des IoC-Prinzips. Spring übernimmt zur Laufzeit die Instanziierung und Zuweisung von Abhängigkeiten. Eine Klasse erhält die @Autowired-Annotation → Spring findet automatisch eine passende Implementierung und injiziert sie.

Es gibt drei Arten der Dependency Injection:

1. Constructor Injection ✓
Best Practice seit Spring 3.5
Abhängigkeiten werden über den Konstruktor übergeben. Das Objekt ist von Beginn an vollständig. 
@Service
public class ControllerImpl {
  private final MovieService svc;

  // @Autowired optional
  public ControllerImpl(
      MovieService svc) {
    this.svc = svc;
  }
}
2. Setter Injection
Abhängigkeiten werden nach der Instanziierung über Setter-Methoden gesetzt. 
@Service
public class MyService {
  private Dep dep;

  @Autowired
  public void setDep(Dep dep) {
    this.dep = dep;
  }
}
3. Field Injection ✗
Direkt auf das Feld annotiert. Nicht empfohlen – schlechte Testbarkeit, versteckte Abhängigkeiten. 
@Service
public class MyService {
  @Autowired
  private Dep dep;
  // nicht empfohlen!
}

Inversion of Control (IoC)

Normalerweise ruft dein Code eine Bibliothek auf. Mit IoC dreht sich das um: Das Framework ruft deinen Code auf. Du gibst die Kontrolle ab.

Konkret: Ohne IoC erstellt Car selbst einen PetrolEngine. Mit IoC bekommt Car irgendeinen Engine von außen – er weiß gar nicht ob es ein Benziner, Diesel oder Elektro ist. Das macht den Code flexibel und testbar.

IoC-Prinzip
Kernparadigma: Objekte sollen so lose gekoppelt (entkoppelt) wie möglich sein.
Grundidee: Klassen nutzen andere Klassen, ohne deren Implementierungsdetails kennen zu müssen. 
// ❌ Ohne IoC – stark gekoppelt (SCHLECHT)
public class Car {
    private PetrolEngine engine = new PetrolEngine(); // direkte Abhängigkeit, schwer austauschbar
}

// ✅ Mit IoC + DI – lose gekoppelt (GUT)
public class Car {
    private Engine engine; // Interface – egal welche Implementierung

    public Car(Engine engine) { // Spring injiziert die passende Implementierung
        this.engine = engine;
    }
}
Vorteile von Dependency Injection:
  • Entkoppelte Weiterentwicklung interagierender Klassen
  • Reduzierte Code-Komplexität durch Schichtentrennung
  • Verbesserte Testbarkeit (lose Kopplung ermöglicht Test-Fassaden / Mocking)
  • Implementierungen zur Laufzeit austauschbar

Component Scan

Beim Anwendungsstart scannt Spring automatisch alle deine Klassen nach Annotationen wie @Service, @Repository etc. Wer eine solche Annotation hat, wird als Bean registriert und verwaltet. Du musst nichts manuell registrieren.

Beim Anwendungsstart führt Spring einen Component Scan durch:
  1. Spring scannt alle Klassen im Classpath nach @Component und Untertypen
  2. Gefundene Klassen werden im Application Context (IoC-Container) als Beans registriert
  3. Abhängigkeiten zwischen Beans werden erkannt und automatisch injiziert
Wichtig: Das Paket mit @SpringBootApplication (inkl. Unterpakete) wird automatisch gescannt. Klassen außerhalb dieses Pakets werden nicht gefunden.

Alle wichtigen Spring-Annotationen (Übersicht)

AnnotationZielBedeutung
@SpringBootApplicationKlasseKombiniert @Configuration + @EnableAutoConfiguration + @ComponentScan. Einstiegspunkt der Anwendung.
@ComponentKlasseGenerischer Spring-Bean. Lebenszyklus von Spring verwaltet.
@ControllerKlassePresentation-Layer. Verarbeitet HTTP-Requests.
@RestControllerKlasseWie @Controller, antwortet automatisch mit JSON.
@ServiceKlasseBusiness-Logic-Layer.
@RepositoryKlasse/InterfaceData-Access-Layer. Datenbankzugriff.
@ConfigurationKlasseKonfigurationsklasse (Configuration as Code).
@AutowiredKonstruktor/Feld/SetterMarkiert Abhängigkeit → Spring injiziert automatisch.
@EventListenerMethodeMethode wird bei bestimmtem Anwendungsevent aufgerufen. Im Projekt: ApplicationReadyEvent → läuft nach dem Start.
@TransactionalKlasse/MethodeMethode/Klasse wird in einem Transaktionskontext ausgeführt.

2 · ORM, JPA & Spring Data

Das Problem: OOP vs. Relationale DB

Java denkt in Objekten (ein Movie-Objekt hat ein Feld director das auf ein Director-Objekt zeigt). Eine Datenbank denkt in Tabellen (die MOVIE-Tabelle hat eine Spalte director_id).

ORM ist der Übersetzer dazwischen. Du arbeitest mit Java-Objekten, ORM übersetzt das automatisch in SQL-Tabellen und zurück.

Objektorientierte Anwendung (OOP)
  • Entitäten als Objekte modelliert
  • Beziehungen über Referenzen/Relationen
  • Klassen mit Verhalten (Methoden)
Relationale Datenbank (SQL)
  • Tabellen im relationalen Schema
  • Beziehungen über Foreign Keys
  • Fokus auf Normalformen & Speichereffizienz

→ Diese grundlegend verschiedenen Paradigmen überbrückt das Object-Relational Mapping (ORM).

ORM (Object-Relational Mapping)

ORM übersetzt zwischen Java-Objekten und relationalen Tabellen:
  • Java-Klassen → Datenbank-Tabellen
  • Klassen-Attribute → Tabellen-Spalten
  • Objekt-Beziehungen → Foreign Keys

ORM-Operationen

OperationBedeutung
createAnwendungsobjekt wird in DB eingefügt (INSERT)
readDB-Datensatz wird zu Anwendungsobjekt konvertiert (SELECT)
updateÄnderungen am Objekt werden in DB synchronisiert (UPDATE)
deleteAnwendungsobjekt wird aus DB gelöscht (DELETE)
Vorteile ORM
  • Harmonisierte DB-Struktur
  • Hoher Automatisierungsgrad
  • Reduzierte Datenkomplexität
  • Schutz vor SQL-Injection
  • Einfacher DB-Wechsel (z. B. H2 → MySQL)
Nachteile ORM
  • DB-Abstraktion kann Performance kosten
  • Initiale Konfiguration komplex
  • SQL-Kenntnisse weiterhin nötig

Glossar: JPA-Technologiestack

JPA ist nur ein Standard (ein Regelwerk, was Annotationen bedeuten sollen). Hibernate ist die eigentliche Implementierung die die Arbeit macht. Spring Data JPA setzt obendrauf noch Komfortfunktionen wie automatische Repositories.

Kurz: JPA = Bauplan, Hibernate = Handwerker, Spring Data JPA = Polier der alles koordiniert.

BegriffErklärung
JPA (Java Persistence API)Spezifikation/Standard für die Beschreibung von Entities und ORM in Java. Definiert Annotationen, Interfaces und Verhalten – keine Implementierung.
Spring Data JPAImplementiert den JPA-Standard und bietet zusätzliche Komfortfunktionen (Repositories, Derived Queries …).
HibernatePopuläres ORM-Framework; die Implementierung hinter Spring Data JPA. Generiert und führt SQL aus.
JDBCLow-Level-Framework für direkten Datenbankzugriff (SQL-Statements manuell).
JDBC-DriverVerbindet JDBC mit der konkreten Datenbank (H2, MySQL, PostgreSQL …).

Entities

Eine Entity ist eine ganz normale Java-Klasse, die du mit @Entity markierst. Damit sagst du Hibernate: "Erstelle für diese Klasse eine Datenbanktabelle und speichere Objekte dieser Klasse darin." Jede Instanz der Klasse = eine Zeile in der Tabelle.

Was ist eine Entity?
Eine Entity ist ein Datenobjekt, das in der Datenbank gespeichert wird. Sie wird mit @Entity markiert und im Paket model abgelegt.

Pflichtanforderungen einer Entity (JPA-Spezifikation):

  1. Leerer Konstruktor (No-Arg Constructor) – Pflicht laut JPA-Spezifikation, damit Hibernate Objekte erstellen kann
  2. Getter & Setter für alle Attribute (manuell oder per Lombok)
  3. Identifikationsschlüssel mit @Id und @GeneratedValue

Entity-Annotationen im Überblick

AnnotationBedeutung
@EntityMarkiert die Klasse als persistierbare Datenbankentität → Hibernate erstellt Tabelle
@IdIdentifiziert das Attribut als Primärschlüssel (eindeutiger Identifikator jedes Datensatzes)
@GeneratedValueSchlüssel wird von Hibernate auto-generiert (1, 2, 3, …). Verhindert Kollisionen.
@ColumnMappt Attribut auf Tabellenspalte (optional, für Umbenennung/Constraints)
@TableDefiniert Tabellenname (optional, Standard = Klassenname)
@JoinColumnDefiniert die Foreign-Key-Spalte für eine Beziehung (z. B. director_id)
@JoinTableDefiniert eine Zwischentabelle (für @ManyToMany)
// Vollständiges Entity-Beispiel aus dem Projekt
@Entity
@Getter @Setter @NoArgsConstructor  // Lombok generiert Getter, Setter und leeren Konstruktor
public class Movie {

    @Id
    @GeneratedValue          // Hibernate vergibt automatisch 1, 2, 3, ...
    private long id;         // Primärschlüssel

    private String title;
    private double duration;         // in Stunden
    private LocalDate premiereDate;  // Datum als Java-Objekt, nicht String
    private String description;
    private String trailerUrl;
    private String coverImgUrl;
    private double imdbRating;
}

Repositories

Ein Repository ist dein Datenbankassistent. Du schreibst nur ein Interface (keine Implementierung!), Spring Data JPA baut die echte Implementierung zur Laufzeit automatisch. Du bekommst sofort save(), findAll(), delete() etc. geschenkt.

Was ist ein Repository?
Ein Repository abstrahiert den Datenbankzugriff für eine Entity. Es enthält alle Funktionen zum Speichern, Abrufen und Löschen von Entities.

No-Code-Ansatz: Nur das Interface muss definiert werden – Spring generiert die Implementierung automatisch.

Ein Repository erweitert JpaRepository<EntityTyp, PrimärschlüsselTyp>:

@Repository
public interface MovieRepository extends JpaRepository<Movie, Long> {
    // Spring implementiert automatisch:
    // save(), findById(), findAll(), delete(), count(), existsById(), ...
}

Mitgelieferte JpaRepository-Methoden

MethodeSQL-ÄquivalentBeschreibung

findAll()SELECT *Alle Entities abrufen

existsById(id)SELECT EXISTSPrüft ob Entity existiert → gibt Boolean zurück

Derived Query Methods (Abgeleitete Anfragen)

Spring liest den Methodennamen und generiert daraus automatisch die SQL-Abfrage. Du schreibst nur den Methodennamen nach dem Namensschema – Spring erledigt den Rest.

Beispiel: findByTitleContaining("Matrix") → Spring generiert SELECT * FROM movie WHERE title LIKE '%Matrix%'

Spring Data JPA kann Repository-Methoden anhand des Methodennamens automatisch implementieren. Keine manuelle SQL/JPQL-Implementierung nötig.

JPQL & @Query-Annotation

Wenn der Methodenname zu komplex wird (z. B. JOINs über mehrere Tabellen), schreibst du die Abfrage manuell mit @Query.

JPQL sieht aus wie SQL, aber: statt Tabellenname schreibst du den Klassennamen, statt Spaltenname das Attribut. Beispiel: statt SELECT * FROM movie schreibst du SELECT m FROM Movie m.

Was ist JPQL?
JPQL (Java Persistence Query Language) ist eine SQL-ähnliche Sprache, die aber auf dem Objektgraphen operiert – d. h. Klassenamen und Attribute statt Tabellennamen und Spalten.
Wichtig: JPQL-Queries werden beim Spring Boot Start validiert. Syntaxfehler fallen sofort beim Start auf – nicht erst zur Laufzeit.
AnnotationBedeutung
@Query("...")Definiert eine manuelle JPQL-Abfrage für eine Repository-Methode
@Param("name")Bindet einen Methodenparameter an einen benannten JPQL-Parameter (:name)
// Einfache JPQL-Abfrage (Aufgabe 2)
// "Movie" = Klassenname, "m" = Alias, "m.title" = Attribut (nicht Spaltenname!)
@Query("SELECT m FROM Movie m WHERE m.title = :param1")
List<Movie> findAllMoviesByTitle(@Param("param1") String title);

// LIKE-Abfrage mit Wildcard (% = beliebige Zeichen)
@Query("SELECT m FROM Movie m WHERE m.title like %:param1%")
List<Movie> findAllMoviesBelongingToSeries(@Param("param1") String series);

// JOIN über eine Collection (ManyToMany)
// "JOIN m.involvedMovieCast c" = verknüpfe mit der Cast-Liste, Alias c
@Query("SELECT m FROM Movie m JOIN m.involvedMovieCast c WHERE c.surname = :param1")
List<Movie> findAllMoviesWithCastSurname(@Param("param1") String castSurname);

// Verschachtelter Attributzugriff über Objekt-Referenz (kein JOIN nötig)
// m.soundtrack.author_surname = über das soundtrack-Attribut auf dessen Felder zugreifen
@Query("SELECT m FROM Movie m WHERE m.soundtrack.author_surname = :param1")
List<Movie> findAllMoviesWithSoundtrackAuthor(@Param("param1") String surname);

// Kollektionsgröße mit size()
@Query("SELECT m FROM Movie m WHERE size(m.involvedMovieCast) > :param1")
List<Movie> findAllMoviesWithMoreThanGivenCasts(@Param("param1") Integer num);

// JOIN über Award-Kollektion (OneToMany)
@Query("SELECT m FROM Movie m JOIN m.awards a WHERE a.academy = :param1")
List<Movie> findAllMoviesWithAcademy(@Param("param1") String academy);

// Mehrere JOINs und Bedingungen kombinieren
@Query("SELECT m FROM Movie m JOIN m.involvedMovieCast c JOIN m.awards a " +
       "WHERE c.surname = :param1 AND a.academy = :param2")
List<Movie> myMoviesByCastAndAward(@Param("param1") String surname,
                                    @Param("param2") String academy);

// Komplexe Abfrage: Award + Datum-Bereich + Kollektionsgröße
@Query("SELECT m FROM Movie m JOIN m.awards a WHERE " +
       "a.category = :param1 AND size(m.involvedMovieCast) > 0 AND " +
       "m.soundtrack.releaseDate > :param2 AND m.soundtrack.releaseDate < :param3")
List<Movie> findMoviesByCastSizeAndSoundtrack(@Param("param1") String category,
                                               @Param("param2") LocalDate minDate,
                                               @Param("param3") LocalDate maxDate);

3 · JPA-Beziehungstypen

Entities stehen in Beziehungen. Das entspricht dem echten Leben: Ein Regisseur dreht mehrere Filme, ein Film hat viele Schauspieler, ein Film hat genau einen Soundtrack.

JPA bildet diese Beziehungen mit Annotationen ab. In der Datenbank werden sie zu Foreign Keys oder Zwischentabellen.

Faustregel: Die Seite die den Foreign Key in der Tabelle hat, ist der Besitzer der Beziehung.

AnnotationKardinalitätBeispielDB-Umsetzung
@ManyToOneN:1Viele Movies → ein DirectorForeign Key in Movie-Tabelle
@OneToMany1:NEin Director → viele MoviesGegenseite zu @ManyToOne (kein extra FK)
@ManyToManyM:NViele Movies ↔ viele SchauspielerZwischentabelle (Join Table)
@OneToOne1:1Ein Movie ↔ ein SoundtrackForeign Key auf einer Seite

@ManyToOne / @OneToMany – 1:N-Beziehung

Ein Director hat viele Movies. Jeder Movie gehört zu einem Director.
In der Datenbank: Die MOVIE-Tabelle bekommt eine Spalte director_id (Foreign Key).

Der Movie ist der "Besitzer" weil er den Foreign Key hat → @ManyToOne beim Movie.
Der Director sieht seine Filme über die Rückseite → @OneToMany(mappedBy = "director") beim Director.

Beispiel: Ein Director hat viele Movies. Jeder Movie hat genau einen Director.
→ Der Movie ist der „Besitzer" der Beziehung (hat den Foreign Key director_id). 
// Movie-Entity (viele Movies → ein Director)  → "@ManyToOne" = Movie-Perspektive: "ich bin Viele"
@ManyToOne(cascade = CascadeType.ALL)
@JoinColumn(name = "director_id", nullable = true)  // erzeugt Spalte "director_id" in MOVIE-Tabelle
private Director director;

// Director-Entity (ein Director → viele Movies)  → "@OneToMany" = Director-Perspektive: "ich bin Einer"
@OneToMany(mappedBy = "director", cascade = CascadeType.ALL)
// mappedBy = Name des Felds auf der Besitzer-Seite (Movie.director)
// → diese Seite hat KEINEN eigenen FK, sie verweist auf die andere Seite
private List<Movie> movies;
mappedBy erklärt: mappedBy = "director" bedeutet: "Die Beziehung ist bereits auf der anderen Seite definiert, im Feld namens director. Ich (Director) bin die nicht-besitzende Seite." Dadurch weiß JPA, dass es keinen zweiten Foreign Key anlegen soll.

@ManyToMany – M:N-Beziehung

Ein Film hat viele Schauspieler, ein Schauspieler spielt in vielen Filmen.
In der Datenbank braucht man dafür eine Zwischentabelle, die einfach nur zwei Foreign Keys enthält: movie_id und cast_id.

Spring legt diese Tabelle automatisch an wenn du @JoinTable definierst.

Beispiel: Viele Movies haben viele Schauspieler (MovieCast). Ein Schauspieler wirkt in mehreren Movies mit.
→ Datenbank benötigt eine Zwischentabelle (movie_moviecast). 
// Movie-Entity (Besitzer-Seite, definiert die Zwischentabelle mit @JoinTable)
@ManyToMany(cascade = CascadeType.ALL)
@JoinTable(
    name = "movie_moviecast",                  // Name der Zwischentabelle in der DB
    joinColumns = {
        @JoinColumn(name = "movie_id",         // FK-Spalte für Movie
                    referencedColumnName = "id")
    },
    inverseJoinColumns = {
        @JoinColumn(name = "cast_id",          // FK-Spalte für MovieCast
                    referencedColumnName = "id")
    }
)
private List<MovieCast> involvedMovieCast;

// MovieCast-Entity (nicht-besitzende Seite, verweist mit mappedBy auf die andere Seite)
@ManyToMany(mappedBy = "involvedMovieCast", cascade = CascadeType.PERSIST)
private List<Movie> participatedMovies;
Wichtig @ManyToMany: Referenz muss immer von der Besitzer-Seite gesetzt werden → movie.getInvolvedMovieCast().add(castMember). Nur dann wird die Zwischentabelle korrekt befüllt.

@OneToOne – 1:1-Beziehung

Ein Film hat genau einen Soundtrack, ein Soundtrack gehört zu genau einem Film.
Hier hat der Soundtrack den Foreign Key (movie_id) in seiner Tabelle → Soundtrack ist Besitzer.

Das ist wie bei einer Ehe: Beide Seiten verweisen aufeinander, aber einer trägt den Eintrag im Register (= FK).

Beispiel: Jeder Movie hat genau einen Soundtrack. Jeder Soundtrack gehört zu genau einem Movie.
→ Soundtrack besitzt die Beziehung (hat den Foreign Key movie_id). 
// Soundtrack-Entity (Besitzer-Seite: hat den FK "movie_id" in seiner Tabelle)
@OneToOne(cascade = CascadeType.ALL)

Cascading (Kaskadierende Operationen)

Cascading bestimmt: wenn ich etwas mit Objekt A mache, was passiert dann automatisch mit Objekt B das mit A verknüpft ist?

Mit CascadeType.ALL beim Movie→Director: wenn du einen Movie speicherst, wird der Director automatisch mitgespeichert. Du musst den Director nicht separat speichern!

Ohne Cascade: du müsstest Director und Movie separat speichern → mehr Code, mehr Fehler.

Cascading definiert, wie sich Operationen an einer Entity auf referenzierte Entities auswirken.
CascadeTypeBedeutung
CascadeType.PERSISTWenn Owner gespeichert wird → referenzierte Entity wird auch gespeichert
CascadeType.REMOVEWenn Owner gelöscht wird → referenzierte Entity wird auch gelöscht
CascadeType.MERGEWenn Owner gemergt (aktualisiert) wird → referenzierte Entity auch
CascadeType.REFRESHWenn Owner aus DB neu geladen wird → referenzierte Entity auch
CascadeType.DETACHWenn Owner vom Context getrennt wird → referenzierte Entity auch
CascadeType.ALLAlle obigen Operationen kaskadieren (PERSIST + REMOVE + MERGE + REFRESH + DETACH)
Cascade-Strategie im Projekt:
CascadeType.ALL bei Movie→Director: Wenn Movie gespeichert wird, wird Director automatisch mitgespeichert.
CascadeType.PERSIST (nicht ALL) bei MovieCast→Movies: Cast wird gespeichert, aber Löschen von Cast löscht nicht automatisch die Movies.

Lazy Loading vs. Eager Loading

Lazy = faul. Spring lädt die verknüpften Objekte erst wenn du wirklich drauf zugreifst (movie.getInvolvedMovieCast() aufrufen). Solange du nicht drauf zugreifst, ist die Liste leer.

Eager = eifrig. Spring lädt alles sofort beim ersten Laden des Movie-Objekts mit – auch wenn du die Cast-Liste vielleicht gar nicht brauchst.

Problem Lazy: Wenn du nach dem Schließen der DB-Session (außerhalb @Transactional) auf eine lazy-geladene Liste zugreifst, gibt es eine LazyInitializationException. Lösung: @Transactional auf die Methode.

Eager Loading
Alle referenzierten Objekte werden sofort beim Laden der Entity aus der DB geladen.

Problem: Kann bei bidirektionalen Beziehungen zu enormen Performance-Problemen führen (zu viele Daten auf einmal).

Default bei: @ManyToOne, @OneToOne
Lazy Loading ✓ (empfohlen)
Referenzierte Objekte werden erst beim Zugriff auf das Attribut geladen.

Empfohlen für Collections (Listen).

Default bei: @OneToMany, @ManyToMany

Achtung: Außerhalb @Transactional → LazyInitializationException! 
// Explizit Eager-Loading setzen (alles immer sofort laden)
@OneToMany(mappedBy = "movie", cascade = CascadeType.ALL, fetch = FetchType.EAGER)
private List<Award> awards;

// Explizit Lazy-Loading (ist Standard bei Collections, kann weggelassen werden)
@ManyToMany(cascade = CascadeType.PERSIST, fetch = FetchType.LAZY)
private List<Movie> participatedMovies;

Bidirektionale Beziehungen

Bei einer bidirektionalen Beziehung kann man von beiden Seiten navigieren: movie.getDirector() und director.getMovies().

Aber: JPA braucht eine klare Aussage, wer den Foreign Key besitzt. Das ist immer die Seite ohne mappedBy. Diese Seite muss immer gesetzt werden – sonst landet nichts in der DB.

Bei bidirektionalen Beziehungen müssen beide Seiten gepflegt werden:
  1. Die Besitzer-Seite (ohne mappedBy) trägt den Foreign Key und steuert die DB-Persistenz
  2. Beim Setzen der Referenz im Code muss immer die Besitzer-Seite gesetzt werden
  3. Für korrekte bidirektionale Navigation im Code sollte auch die andere Seite gesetzt werden
// Korrekte bidirektionale Beziehungspflege
Director director = new Director("Christopher", "Nolan");
Movie movie = new Movie("Interstellar");

// 1. Besitzer-Seite setzen (Movie hat den FK) → Pflicht für die DB!
movie.setDirector(director);

// 2. Gegenseite setzen → Pflicht für korrekte Java-Navigation (director.getMovies() liefert sonst leer)
director.getMovies().add(movie);

// 3. Nur Movie speichern – Director wird durch CascadeType.ALL automatisch mitgespeichert!
movieRepository.save(movie);
// KEIN direktory.save() nötig

4 · Transaktionen

Eine Transaktion ist wie ein "Alles oder Nichts"-Versprechen an die Datenbank. Stell dir vor du überweist Geld: Konto A wird belastet UND Konto B wird gutgeschrieben. Wenn der zweite Schritt fehlschlägt, muss auch der erste rückgängig gemacht werden (Rollback). Genau das macht eine Transaktion.

Was ist eine Transaktion?
Eine Transaktion fasst zusammengehörige Operationen zu einer Einheit zusammen. Sie implementiert das Alles-oder-Nichts-Prinzip: Entweder alle Operationen gelingen, oder bei einem Fehler wird alles rückgängig gemacht (Rollback).

ACID-Eigenschaften

ACID beschreibt die vier Garantien die eine Transaktion geben muss. Lerne das Akronym: Atomarität, Consistenz (Konsistenz), Isolation, Durability (Dauerhaftigkeit).

A – Atomicity (Atomarität)
Eine Transaktion ist nur erfolgreich, wenn alle Operationen erfolgreich sind.

Bei Fehler → automatischer Rollback zum Ausgangszustand.

Einfach: Entweder alles klappt, oder gar nichts ändert sich.
C – Consistency (Konsistenz)
Die Transaktion beginnt und endet in einem konsistenten DB-Zustand.

Alle Constraints/Bedingungen bleiben erhalten (z.B. NOT NULL, Foreign Keys).

Einfach: Die Datenbank bleibt immer in einem gültigen Zustand.
I – Isolation
Transaktion läuft unabhängig von parallel laufenden Transaktionen.

Keine Seiteneffekte durch parallele Transaktionen.

Einfach: Zwei gleichzeitige Transaktionen stören sich nicht gegenseitig.
D – Durability (Dauerhaftigkeit)
Ergebnisse einer erfolgreichen Transaktion sind permanent gespeichert und überleben Server-Abstürze.

Einfach: Was einmal committed ist, bleibt auch nach einem Stromausfall erhalten.

@Transactional Annotation

@Transactional auf einer Methode bedeutet: Spring öffnet vor dem Methodenaufruf eine DB-Session, und schließt sie (mit Commit) danach. Innerhalb der Methode:
1. Du kannst lazy-geladene Collections problemlos aufrufen (Session ist noch offen).
2. Wenn du ein Objekt änderst (setTitle(...)), speichert Spring die Änderung automatisch am Ende – kein save() nötig! (= Dirty Checking)

@Transactional markiert eine Methode oder Klasse als transaktional. Spring öffnet automatisch einen Transaktionskontext vor der Methode und committed/rollt zurück nach der Methode.
AnwendungsortWirkung
Auf MethodeNur diese Methode läuft transaktional (empfohlen – gezielter Einsatz)
Auf KlasseAlle Methoden der Klasse laufen transaktional
@Service
public class MovieServiceImpl implements MovieService {

    private MovieRepository movieRepository;

    // Constructor Injection
    public MovieServiceImpl(MovieRepository movieRepository) {
        this.movieRepository = movieRepository;
    }

    @Transactional  // Session bleibt offen → Lazy-Loading klappt!
    public List<String> getCastOfMovie(String movieTitle) {
        List<String> result = new LinkedList<>();
        List<Movie> tmp = this.movieRepository.findByTitleContaining(movieTitle);
        if (tmp.size() == 1) {
            // involvedMovieCast ist lazy → würde OHNE @Transactional
            // eine LazyInitializationException werfen!
            tmp.get(0).getInvolvedMovieCast().forEach(cast ->
                result.add(cast.getForename())
            );
        }
        return result;
    }

    @Transactional  // Dirty Checking: kein save() nötig!
    public void changeMovieTitle(long id, String newMovieName) {
        if (!newMovieName.isEmpty()) {
            Movie tmp = this.movieRepository.findById(id).get();
            tmp.setTitle(newMovieName);
            // Hibernate erkennt die Änderung automatisch und führt UPDATE aus
            // KEIN movieRepository.save(tmp) nötig innerhalb @Transactional!
        }
    }
}
Dirty Checking erklärt: Innerhalb einer Transaktion merkt sich Hibernate den Zustand aller geladenen Objekte. Beim Commit vergleicht es den aktuellen Zustand mit dem Anfangszustand und führt automatisch UPDATEs für alle geänderten Objekte aus.
Standard-Transaktionen: Alle CRUD-Operationen des JpaRepository (save, findById, findAll, delete) führen bereits automatisch als Transaktionen aus.

Aufgabe 3 – Beziehungen korrekt setzen Aufgabe 3

Hier siehst du wie man alle Beziehungen im Code korrekt verknüpft. Wichtig: Immer die Besitzer-Seite setzen, und nur movieRepository.save(movie) aufrufen – Cascade kümmert sich um den Rest.

@Component
@Slf4j
public class ManualTestBean {

    private MovieRepository movieRepository;
    private MovieService movieService;

    public ManualTestBean(MovieRepository movieRepository, MovieService movieService) {
        this.movieRepository = movieRepository;
        this.movieService = movieService;
    }

    @EventListener(ApplicationReadyEvent.class)
    public void runAfterStartup() {

        // Entities erstellen
        Director nolan = new Director();
        nolan.setForename("Christopher"); nolan.setSurname("Nolan");

        MovieCast actor = new MovieCast();
        actor.setForename("Matthew"); actor.setSurname("McConaughey");

        Soundtrack st = new Soundtrack();
        st.setName("Interstellar OST");
        st.setAuthor_forename("Hans"); st.setAuthor_surname("Zimmer");

        Movie interstellar = new Movie();
        interstellar.setTitle("Interstellar");
        interstellar.setImdbRating(8.7);

        // ── Beziehungen setzen ────────────────────────────────
        // 1. N:1: Movie → Director
        interstellar.setDirector(nolan);             // Besitzer-Seite (Movie hat FK)
        nolan.setMovies(new ArrayList<>(List.of(interstellar))); // Gegenseite

        // 2. M:N: Movie ↔ MovieCast
        interstellar.setInvolvedMovieCast(new ArrayList<>(List.of(actor))); // Besitzer-Seite
        actor.setParticipatedMovies(new ArrayList<>(List.of(interstellar))); // Gegenseite

        // 3. 1:1: Movie ↔ Soundtrack (Soundtrack ist Besitzer!)
        st.setMovie(interstellar);                   // Besitzer-Seite (Soundtrack hat FK)
        interstellar.setSoundtrack(st);              // Gegenseite

        // ── Speichern ──────────────────────────────────────────
        // Nur Movie speichern! CascadeType.ALL kümmert sich um Director, Cast, Soundtrack
        movieRepository.save(interstellar);

        // ── Abfragen demonstrieren ─────────────────────────────
        log.info("Alle Filme: {}",
            movieRepository.findAll().stream().map(Movie::getTitle).toList());
        log.info("Filme mit 'Inter': {}",
            movieRepository.findByTitleContaining("Inter"));
        log.info("Cast von Interstellar: {}",
            movieService.getCastOfMovie("Interstellar"));  // @Transactional in Service!
        log.info("Filme mit Zimmer-Soundtrack: {}",
            movieRepository.findAllMoviesWithSoundtrackAuthor("Zimmer"));
    }
}
