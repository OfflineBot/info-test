1 · Spring Grundlagen

Spring ist ein riesiger Werkzeugkasten für Java-Entwickler. Statt alles selbst zu verdrahten (wer ruft wen auf? wer erstellt welches Objekt?), übernimmt Spring das für dich. Du sagst Spring was du brauchst, Spring kümmert sich um das Wie.

Spring ist ein Open-Source-Framework zur plattformunabhängigen Entwicklung von Web-Applikationen mit Java. Es basiert auf vier Kernprinzipien:

Convention over Configuration
Best-Practice-Annahmen sind voreingestellt; Entwickler spezifizieren nur Abweichungen vom Standard. Weniger Konfigurationsaufwand.
Modularität
Spring ist in Module aufgeteilt, die nur bei Bedarf geladen werden (Web, Data, Security …).
Schichtenarchitektur
Klare Trennung von Presentation, Business Logic und Data Storage.
Dependency Injection
Spring übernimmt die Verwaltung von Objekten, um Komplexität zu reduzieren.
Spring Boot Baut auf dem Spring Framework auf und nutzt Standardkonfigurationen, um selbstständig ausführbare Web-Applikationen als JAR-Datei schnell zu entwickeln. Früher war ein dedizierter Applikationsserver (z. B. Tomcat) nötig – Spring Boot bringt ihn bereits mit.

Spring Boot ist Spring mit sinnvollen Voreinstellungen. Du startest eine fertige App mit einem einzigen java -jar app.jar – kein extra Tomcat installieren nötig.

Schichtenarchitektur

Die App ist in Schichten aufgeteilt, damit jede Klasse genau eine Aufgabe hat. Der Controller redet mit dem Browser, der Service denkt nach, das Repository spricht mit der Datenbank – keiner mischt sich in die Arbeit des anderen.

Spring Boot strukturiert Anwendungen in vier funktionale Schichten:

LayerAnnotationAufgabe
Controller @Controller / @RestController Verarbeitet Client-Anfragen (HTTP-Requests) und gibt Antworten zurück (HTML, JSON …)
Service @Service Kapselt und implementiert die Geschäftslogik
Repository @Repository Verwaltet Datenzugriff und -speicherung (Datenbankoperationen)
Entity / Model @Entity Repräsentiert Datenbankobjekte (Tabellen)
Älteres 3-Schichten-Modell (Grundprinzip)
  • Presentation Layer – Anzeige & Eingabe (Frontend)
  • Business Logic Layer – Geschäftslogik
  • Data Storage Layer – Datenspeicherung & Datenbankzugriff

Spring Beans

Normalerweise erstellst du Objekte mit new MeineKlasse(). Bei Spring machst du das nicht mehr selbst – du markierst die Klasse mit einer Annotation und Spring erstellt und verwaltet das Objekt für dich. Dieses von Spring verwaltete Objekt heißt Bean.

Vergleich: Du gehst ins Restaurant. Du bestellst (= Annotation) und der Kellner (= Spring) bringt dir das Essen (= Objekt). Du kochst nicht selbst.

Was ist ein Bean?
Ein Bean ist eine Java-Klasse, deren Lebenszyklus (Erstellung, Konfiguration, Skalierung, Löschung) vollständig vom Spring-Framework verwaltet wird.

Beans werden nicht manuell per new instanziiert – Spring übernimmt das im sogenannten IoC-Container (Application Context). 
@Service
public class MovieServiceImpl implements MovieService {
    // Spring erstellt und verwaltet diese Klasse automatisch
    // Kein "new MovieServiceImpl()" irgendwo nötig!
}

Stereotype-Annotationen (Bean-Typen)

Stereotypen sind wie Berufsbezeichnungen für deine Klassen. @Service = Buchhalter (Geschäftslogik), @Repository = Archivar (Datenbank), @Controller = Rezeption (HTTP-Anfragen).

Alle sind letztendlich @Component – das ist der Oberbegriff für "von Spring verwaltetes Objekt".

@Component ist die Basis-Annotation. Alle anderen Stereotypen sind spezialisierte Untertypen:

@Component – generischer Spring-Bean (keine Schichtzuordnung)
├── @Controller – verarbeitet Client-Anfragen / gibt Antworten zurück
│   └── @RestController – wie @Controller, Ausgabe als JSON
├── @Service – implementiert Geschäftslogik
├── @Repository – Datenbankzugriff & -verwaltung
└── @Configuration – Infrastruktur-Konfiguration ("Configuration as Code")
AnnotationSchichtZweck & Details
@ComponentGenerischBasis für alle Stereotypen. Markiert eine Klasse als Spring-verwalteten Bean. Kein expliziter Layer.
@ControllerPresentationKommuniziert mit dem Benutzer/API. Nimmt HTTP-Requests entgegen, gibt Antworten in verschiedenen Formaten zurück.
@RestControllerPresentationUnterklasse von @Controller. Gibt automatisch JSON zurück (kombiniert @Controller + @ResponseBody). Für REST-APIs.
@ServiceBusiness LogicKapselt Geschäftslogik. Wird von Controllern aufgerufen. Kein direkter Datenbankzugriff.
@RepositoryData AccessDatenspeicherung und -verwaltung. Spring übersetzt DB-Exceptions automatisch in Spring-Exceptions.
@ConfigurationInfrastruktur"Configuration as Code"-Paradigma. Konfiguriert Spring-Infrastruktur programmatisch.

Dependency Injection (DI)

Stell dir vor, ein Auto baut sich seinen Motor selbst – das wäre schlechtes Design. Besser: Der Motor wird dem Auto von außen gegeben. Das ist Dependency Injection.

In Java: Eine Klasse bekommt ihre Abhängigkeiten von Spring automatisch "injiziert" – sie baut sie nicht selbst mit new.

Was ist Dependency Injection?
DI ist eine Implementierung des IoC-Prinzips. Spring übernimmt zur Laufzeit die Instanziierung und Zuweisung von Abhängigkeiten. Eine Klasse erhält die @Autowired-Annotation → Spring findet automatisch eine passende Implementierung und injiziert sie.

Es gibt drei Arten der Dependency Injection:

1. Constructor Injection ✓
Best Practice seit Spring 3.5
Abhängigkeiten werden über den Konstruktor übergeben. Das Objekt ist von Anfang an vollständig. 
@Service
public class CtrlImpl {
  private final MovieService s;

  // @Autowired optional
  public CtrlImpl(MovieService s) {
    this.s = s;
  }
}
2. Setter Injection
Abhängigkeiten werden nach der Instanziierung über Setter gesetzt. 
@Service
public class MyService {
  private Dep dep;

  @Autowired
  public void setDep(Dep d) {
    this.dep = d;
  }
}
3. Field Injection ✗
Direkt auf das Feld. Nicht empfohlen – schlechte Testbarkeit. 
@Service
public class MyService {
  @Autowired
  private Dep dep;
  // nicht empfohlen!
}

Inversion of Control (IoC)

Normalerweise ruft dein Code eine Bibliothek auf. Mit IoC dreht sich das um: Das Framework ruft deinen Code auf. Du gibst die Kontrolle ab.

Konkret: Ohne IoC erstellt Car selbst einen PetrolEngine. Mit IoC bekommt Car irgendeinen Engine von außen – er weiß gar nicht ob es ein Benziner, Diesel oder Elektro ist. Das macht den Code flexibel und testbar.

// ❌ Ohne IoC – stark gekoppelt (SCHLECHT)
public class Car {
    private PetrolEngine engine = new PetrolEngine(); // direkte Abhängigkeit, schwer austauschbar
}

// ✅ Mit IoC + DI – lose gekoppelt (GUT)
public class Car {
    private Engine engine; // Interface – egal welche Implementierung

    public Car(Engine engine) { // Spring injiziert die passende Implementierung
        this.engine = engine;
    }
}
Vorteile von Dependency Injection:
  • Entkoppelte Weiterentwicklung interagierender Klassen
  • Reduzierte Code-Komplexität durch Schichtentrennung
  • Verbesserte Testbarkeit (lose Kopplung ermöglicht Mocking)
  • Implementierungen zur Laufzeit austauschbar

Component Scan

Beim Anwendungsstart scannt Spring automatisch alle Klassen nach Annotationen wie @Service, @Repository etc. Wer eine solche Annotation hat, wird als Bean registriert. Du musst nichts manuell registrieren.

Beim Anwendungsstart führt Spring einen Component Scan durch:
  1. Spring scannt alle Klassen im Classpath nach @Component und Untertypen
  2. Gefundene Klassen werden im Application Context (IoC-Container) als Beans registriert
  3. Abhängigkeiten zwischen Beans werden erkannt und automatisch injiziert
Wichtig: Das Paket mit @SpringBootApplication (inkl. Unterpakete) wird automatisch gescannt. Klassen außerhalb werden nicht gefunden.

Alle wichtigen Spring-Annotationen

AnnotationZielBedeutung
@SpringBootApplicationKlasseKombiniert @Configuration + @EnableAutoConfiguration + @ComponentScan. Einstiegspunkt der Anwendung.
@ComponentKlasseGenerischer Spring-Bean. Lebenszyklus von Spring verwaltet.
@ControllerKlassePresentation-Layer. Verarbeitet HTTP-Requests.
@RestControllerKlasseWie @Controller, antwortet automatisch mit JSON.
@ServiceKlasseBusiness-Logic-Layer.
@RepositoryKlasse/InterfaceData-Access-Layer. Datenbankzugriff.
@ConfigurationKlasseKonfigurationsklasse (Configuration as Code).
@AutowiredKonstruktor/Feld/SetterMarkiert Abhängigkeit → Spring injiziert automatisch.
@EventListenerMethodeMethode wird bei Anwendungsevent aufgerufen. Im Projekt: ApplicationReadyEvent → läuft nach dem Start.
@TransactionalKlasse/MethodeMethode/Klasse wird in einem Transaktionskontext ausgeführt.

2 · ORM, JPA & Spring Data

Das Problem: OOP vs. Relationale DB

Java denkt in Objekten (ein Movie-Objekt hat ein Feld director das auf ein Director-Objekt zeigt). Eine Datenbank denkt in Tabellen (die MOVIE-Tabelle hat eine Spalte director_id).

ORM ist der Übersetzer dazwischen. Du arbeitest mit Java-Objekten, ORM übersetzt das automatisch in SQL und zurück.

Objektorientierte Anwendung (OOP)
  • Entitäten als Objekte modelliert
  • Beziehungen über Referenzen/Relationen
  • Klassen mit Verhalten (Methoden)
Relationale Datenbank (SQL)
  • Tabellen im relationalen Schema
  • Beziehungen über Foreign Keys
  • Fokus auf Normalformen & Speichereffizienz

ORM (Object-Relational Mapping)

ORM übersetzt zwischen Java-Objekten und relationalen Tabellen:
  • Java-Klassen → Datenbank-Tabellen
  • Klassen-Attribute → Tabellen-Spalten
  • Objekt-Beziehungen → Foreign Keys
ORM-OperationBedeutung
createAnwendungsobjekt wird in DB eingefügt (INSERT)
readDB-Datensatz wird zu Anwendungsobjekt konvertiert (SELECT)
updateÄnderungen am Objekt werden in DB synchronisiert (UPDATE)
deleteAnwendungsobjekt wird aus DB gelöscht (DELETE)
Vorteile ORM
  • Harmonisierte DB-Struktur
  • Hoher Automatisierungsgrad
  • Schutz vor SQL-Injection
  • Einfacher DB-Wechsel (H2 → MySQL)
Nachteile ORM
  • DB-Abstraktion kann Performance kosten
  • Initiale Konfiguration komplex
  • SQL-Kenntnisse weiterhin nötig

Glossar: JPA-Technologiestack

JPA ist nur ein Standard (ein Regelwerk). Hibernate ist die eigentliche Implementierung die die Arbeit macht. Spring Data JPA setzt obendrauf noch Komfortfunktionen.

Kurz: JPA = Bauplan, Hibernate = Handwerker, Spring Data JPA = Polier der alles koordiniert.

BegriffErklärung
JPASpezifikation/Standard für ORM in Java. Definiert Annotationen und Interfaces – keine Implementierung.
Spring Data JPAImplementiert den JPA-Standard + zusätzliche Komfortfunktionen (Repositories, Derived Queries …).
HibernatePopuläres ORM-Framework; die eigentliche Implementierung hinter Spring Data JPA. Generiert und führt SQL aus.
JDBCLow-Level-Framework für direkten Datenbankzugriff (SQL manuell).
JDBC-DriverVerbindet JDBC mit der konkreten Datenbank (H2, MySQL, PostgreSQL …).

Entities

Eine Entity ist eine ganz normale Java-Klasse mit @Entity. Damit sagst du Hibernate: "Erstelle für diese Klasse eine Datenbanktabelle." Jede Instanz der Klasse = eine Zeile in der Tabelle.

Pflichtanforderungen einer Entity (JPA-Spezifikation):

  1. Leerer Konstruktor (No-Arg Constructor) – Hibernate braucht ihn um Objekte zu erstellen
  2. Getter & Setter für alle Attribute (manuell oder per Lombok)
  3. Identifikationsschlüssel mit @Id und @GeneratedValue
AnnotationBedeutung
@EntityMarkiert die Klasse als persistierbare Datenbankentität → Hibernate erstellt Tabelle
@IdIdentifiziert das Attribut als Primärschlüssel
@GeneratedValueSchlüssel wird von Hibernate auto-generiert (1, 2, 3, …)
@ColumnMappt Attribut auf Tabellenspalte (optional, für Umbenennung/Constraints)
@TableDefiniert Tabellenname (optional, Standard = Klassenname)
@JoinColumnDefiniert die Foreign-Key-Spalte für eine Beziehung (z. B. director_id)
@JoinTableDefiniert eine Zwischentabelle (für @ManyToMany)
@Entity
@Getter @Setter @NoArgsConstructor  // Lombok generiert Getter, Setter und leeren Konstruktor
public class Movie {

    @Id
    @GeneratedValue          // Hibernate vergibt automatisch 1, 2, 3, ...
    private long id;

    private String title;
    private double duration;
    private LocalDate premiereDate;
    private String description;
    private String trailerUrl;
    private String coverImgUrl;
    private double imdbRating;
}

Repositories

Ein Repository ist dein Datenbankassistent. Du schreibst nur ein Interface (keine Implementierung!), Spring Data JPA baut die echte Implementierung automatisch. Du bekommst sofort save(), findAll(), delete() etc. geschenkt.

Was ist ein Repository?
Ein Repository abstrahiert den Datenbankzugriff. No-Code-Ansatz: Nur das Interface muss definiert werden – Spring generiert die Implementierung automatisch. 
@Repository
public interface MovieRepository extends JpaRepository {
    // Spring implementiert automatisch:
    // save(), findById(), findAll(), delete(), count(), existsById(), ...
}
MethodeSQL-ÄquivalentBeschreibung
save(entity)INSERT / UPDATEEntity speichern oder aktualisieren
findById(id)SELECT WHERE id=?Entity per Primärschlüssel laden → gibt Optional<T> zurück
findAll()SELECT *Alle Entities abrufen
delete(entity)DELETEEntity löschen
deleteById(id)DELETE WHERE id=?Entity per ID löschen
count()SELECT COUNT(*)Anzahl der Einträge
existsById(id)SELECT EXISTSPrüft ob Entity existiert

JPQL & @Query

Wenn der Methodenname zu komplex wird (z. B. JOINs), schreibst du die Abfrage manuell mit @Query.

JPQL sieht aus wie SQL, aber: statt Tabellenname schreibst du den Klassennamen, statt Spaltenname das Attribut. Statt SELECT * FROM movie schreibst du SELECT m FROM Movie m.

AnnotationBedeutung
@Query("...")Definiert eine manuelle JPQL-Abfrage für eine Repository-Methode
@Param("name")Bindet einen Methodenparameter an einen benannten JPQL-Parameter (:name)
Wichtig: JPQL-Queries werden beim Spring Boot Start validiert. Syntaxfehler fallen sofort auf – nicht erst zur Laufzeit. 
// Einfache JPQL – "Movie" = Klassenname, "m" = Alias, "m.title" = Attribut
@Query("SELECT m FROM Movie m WHERE m.title = :param1")
List findAllMoviesByTitle(@Param("param1") String title);

// LIKE mit Wildcard (% = beliebige Zeichen)
@Query("SELECT m FROM Movie m WHERE m.title like %:param1%")
List findAllMoviesBelongingToSeries(@Param("param1") String series);

// JOIN über Collection (ManyToMany) – "JOIN m.involvedMovieCast c"
@Query("SELECT m FROM Movie m JOIN m.involvedMovieCast c WHERE c.surname = :param1")
List findAllMoviesWithCastSurname(@Param("param1") String castSurname);

// Verschachtelter Attributzugriff (kein JOIN nötig)
@Query("SELECT m FROM Movie m WHERE m.soundtrack.author_surname = :param1")
List findAllMoviesWithSoundtrackAuthor(@Param("param1") String surname);

// Kollektionsgröße mit size()
@Query("SELECT m FROM Movie m WHERE size(m.involvedMovieCast) > :param1")
List findAllMoviesWithMoreThanGivenCasts(@Param("param1") Integer num);

// JOIN über Award-Kollektion
@Query("SELECT m FROM Movie m JOIN m.awards a WHERE a.academy = :param1")
List findAllMoviesWithAcademy(@Param("param1") String academy);

// Mehrere JOINs kombiniert
@Query("SELECT m FROM Movie m JOIN m.involvedMovieCast c JOIN m.awards a " +
       "WHERE c.surname = :param1 AND a.academy = :param2")
List myMoviesByCastAndAward(@Param("param1") String surname,
                                    @Param("param2") String academy);

3 · JPA-Beziehungstypen

Entities stehen in Beziehungen. Das entspricht dem echten Leben: Ein Regisseur dreht mehrere Filme, ein Film hat viele Schauspieler, ein Film hat genau einen Soundtrack.

Faustregel: Die Seite die den Foreign Key in der Tabelle hat, ist der Besitzer der Beziehung. Die andere Seite hat mappedBy.

AnnotationKardinalitätBeispielDB-Umsetzung
@ManyToOneN:1Viele Movies → ein DirectorForeign Key in Movie-Tabelle
@OneToMany1:NEin Director → viele MoviesGegenseite (kein eigener FK)
@ManyToManyM:NViele Movies ↔ viele SchauspielerZwischentabelle (Join Table)
@OneToOne1:1Ein Movie ↔ ein SoundtrackForeign Key auf einer Seite

@ManyToOne / @OneToMany – 1:N-Beziehung

Ein Director hat viele Movies. Jeder Movie gehört zu einem Director.
In der DB: Die MOVIE-Tabelle bekommt eine Spalte director_id (Foreign Key) → Movie ist Besitzer → @ManyToOne beim Movie.
Der Director sieht seine Filme über die Rückseite → @OneToMany(mappedBy = "director").

// Movie-Entity: "viele Movies → ein Director" → @ManyToOne (Movie-Perspektive: "ich bin Viele")
@ManyToOne(cascade = CascadeType.ALL)
@JoinColumn(name = "director_id", nullable = true)  // erzeugt Spalte "director_id" in MOVIE
private Director director;

// Director-Entity: Rückseite – kein eigener FK, verweist mit mappedBy auf Movie.director
@OneToMany(mappedBy = "director", cascade = CascadeType.ALL)
private List movies;
mappedBy erklärt: mappedBy = "director" bedeutet: "Die Beziehung ist bereits auf der anderen Seite definiert, im Feld director der Movie-Klasse. Ich (Director) bin die nicht-besitzende Seite – JPA legt keinen zweiten FK an."

@ManyToMany – M:N-Beziehung

Ein Film hat viele Schauspieler, ein Schauspieler spielt in vielen Filmen.
In der DB braucht man eine Zwischentabelle mit zwei Foreign Keys: movie_id und cast_id. Spring legt sie automatisch an wenn du @JoinTable definierst.

// Movie-Entity: Besitzer-Seite – definiert die Zwischentabelle mit @JoinTable
@ManyToMany(cascade = CascadeType.ALL)
@JoinTable(
    name = "movie_moviecast",                  // Name der Zwischentabelle in der DB
    joinColumns = {
        @JoinColumn(name = "movie_id", referencedColumnName = "id")
    },
    inverseJoinColumns = {
        @JoinColumn(name = "cast_id",  referencedColumnName = "id")
    }
)
private List involvedMovieCast;

// MovieCast-Entity: nicht-besitzende Seite
@ManyToMany(mappedBy = "involvedMovieCast", cascade = CascadeType.PERSIST)
private List participatedMovies;
Wichtig: Referenz immer von der Besitzer-Seite setzen → movie.getInvolvedMovieCast().add(castMember). Nur dann wird die Zwischentabelle korrekt befüllt.

@OneToOne – 1:1-Beziehung

Ein Film hat genau einen Soundtrack, ein Soundtrack gehört zu genau einem Film.
Hier hat der Soundtrack den Foreign Key (movie_id) → Soundtrack ist Besitzer.
Typische Falle: beide Seiten müssen gesetzt werden bevor man speichert!

// Soundtrack: Besitzer-Seite – hat FK "movie_id" in seiner Tabelle
@OneToOne(cascade = CascadeType.ALL)
@JoinColumn(name = "movie_id")
private Movie movie;

// Movie: nicht-besitzende Seite – kein FK, nur Rückreferenz
@OneToOne(mappedBy = "movie", cascade = CascadeType.ALL)
private Soundtrack soundtrack;
Typische Falle @OneToOne: Beide Seiten müssen gesetzt werden! 
soundtrack.setMovie(movie);         // Besitzer-Seite (Pflicht für DB!)
movie.setSoundtrack(soundtrack);    // Gegenseite (für Java-Navigation)
movieRepository.save(movie);        // Cascade speichert Soundtrack automatisch

Weitere Beziehung: Movie ↔ Award (1:N)

// Movie-Entity (nicht-besitzende Seite, EAGER weil immer mitgeladen)
@OneToMany(mappedBy = "movie", cascade = CascadeType.ALL, fetch = FetchType.EAGER)
private List awards;

// Award-Entity (Besitzer-Seite: hat FK "movie_id")
@ManyToOne(cascade = CascadeType.ALL)
@JoinColumn(name = "movie_id")
private Movie movie;

Cascading

Cascade bestimmt: wenn ich etwas mit Objekt A mache, was passiert automatisch mit Objekt B?

Mit CascadeType.ALL beim Movie→Director: wenn du einen Movie speicherst, wird der Director automatisch mitgespeichert. Du musst ihn nicht separat speichern!

CascadeTypeBedeutung
PERSISTWenn Owner gespeichert wird → referenzierte Entity wird auch gespeichert
REMOVEWenn Owner gelöscht wird → referenzierte Entity wird auch gelöscht
MERGEWenn Owner gemergt wird → referenzierte Entity auch
REFRESHWenn Owner neu geladen wird → referenzierte Entity auch
DETACHWenn Owner vom Context getrennt wird → referenzierte Entity auch
ALLAlle obigen Operationen kaskadieren
Im Projekt: CascadeType.ALL bei Movie→Director: Movie speichern = Director automatisch mitgespeichert.
CascadeType.PERSIST (nicht ALL) bei MovieCast: Cast wird gespeichert, aber Löschen von Cast löscht nicht automatisch Movies.

Lazy Loading vs. Eager Loading

Lazy = faul. Spring lädt verknüpfte Objekte erst wenn du wirklich drauf zugreifst. Solange du nicht zugreifst, ist die Liste ein Platzhalter.

Eager = eifrig. Spring lädt alles sofort beim Laden des Haupt-Objekts – auch wenn du die Liste vielleicht gar nicht brauchst.

Problem Lazy: Außerhalb einer Transaktion (@Transactional) auf lazy-geladene Liste zugreifen → LazyInitializationException!

Eager Loading
Alle referenzierten Objekte werden sofort beim Laden aus der DB mitgeladen.

Problem: Kann bei tiefen Beziehungen enorme Performance-Probleme verursachen.

Default bei: @ManyToOne, @OneToOne
Lazy Loading ✓ (empfohlen)
Referenzierte Objekte werden erst beim Zugriff auf das Attribut geladen.

Empfohlen für Collections (Listen).

Default bei: @OneToMany, @ManyToMany

Achtung: Außerhalb @Transactional → Exception! 
// Explizit Eager setzen
@OneToMany(mappedBy = "movie", cascade = CascadeType.ALL, fetch = FetchType.EAGER)
private List awards;

// Explizit Lazy (Standard bei Collections, kann weggelassen werden)
@ManyToMany(cascade = CascadeType.PERSIST, fetch = FetchType.LAZY)
private List participatedMovies;

Bidirektionale Beziehungen

Bei einer bidirektionalen Beziehung kann man von beiden Seiten navigieren: movie.getDirector() und director.getMovies().

Aber: JPA braucht eine klare Aussage, wer den Foreign Key besitzt (= Seite ohne mappedBy). Diese Seite muss immer gesetzt werden – sonst landet nichts in der DB.

Director director = new Director("Christopher", "Nolan");
Movie movie = new Movie("Interstellar");

// 1. Besitzer-Seite setzen (Movie hat den FK) → PFLICHT für die DB!
movie.setDirector(director);

// 2. Gegenseite setzen → für korrekte Java-Navigation (sonst liefert director.getMovies() leer)
director.getMovies().add(movie);

// 3. Nur Movie speichern – Director durch CascadeType.ALL automatisch mitgespeichert
movieRepository.save(movie);
// KEIN director.save() nötig!

4 · Transaktionen

Eine Transaktion ist ein "Alles oder Nichts"-Versprechen. Beispiel Banküberweisung: Konto A belastet UND Konto B gutgeschrieben. Wenn Schritt 2 fehlschlägt, muss Schritt 1 rückgängig gemacht werden (Rollback). Genau das macht eine Transaktion.

Was ist eine Transaktion?
Eine Transaktion fasst zusammengehörige Operationen zu einer Einheit zusammen. Sie implementiert das Alles-oder-Nichts-Prinzip: Entweder alle Operationen gelingen, oder bei einem Fehler wird alles rückgängig gemacht (Rollback).

ACID-Eigenschaften

A – Atomicity (Atomarität)
Transaktion ist nur erfolgreich wenn alle Operationen erfolgreich sind.
Bei Fehler → Rollback zum Ausgangszustand.

Einfach: Entweder alles klappt, oder gar nichts ändert sich.
C – Consistency (Konsistenz)
Transaktion beginnt und endet in einem konsistenten DB-Zustand.
Alle Constraints bleiben erhalten (NOT NULL, Foreign Keys …).

Einfach: Die DB bleibt immer in einem gültigen Zustand.
I – Isolation
Transaktion läuft unabhängig von parallel laufenden Transaktionen.
Keine Seiteneffekte durch Parallelität.

Einfach: Zwei gleichzeitige Transaktionen stören sich nicht.
D – Durability (Dauerhaftigkeit)
Ergebnisse einer erfolgreichen Transaktion sind permanent gespeichert und überleben Server-Abstürze.

Einfach: Was committed ist, bleibt nach Stromausfall erhalten.

@Transactional

@Transactional auf einer Methode bedeutet: Spring öffnet vor dem Aufruf eine DB-Session und schließt sie (mit Commit) danach. Dadurch:

  • Lazy-geladene Collections können problemlos aufgerufen werden (Session ist noch offen)
  • Wenn du ein Objekt änderst (setTitle(...)), speichert Spring die Änderung automatisch – kein save() nötig! (= Dirty Checking)
AnwendungsortWirkung
Auf MethodeNur diese Methode läuft transaktional (empfohlen)
Auf KlasseAlle Methoden der Klasse laufen transaktional
@Service
public class MovieServiceImpl implements MovieService {

    private MovieRepository movieRepository;

    public MovieServiceImpl(MovieRepository movieRepository) {
        this.movieRepository = movieRepository;
    }

    @Transactional  // Session bleibt offen → Lazy-Loading klappt!
    public List getCastOfMovie(String movieTitle) {
        List result = new LinkedList<>();
        List tmp = this.movieRepository.findByTitleContaining(movieTitle);
        if (tmp.size() == 1) {
            // involvedMovieCast ist lazy-geladen → OHNE @Transactional: LazyInitializationException!
            tmp.get(0).getInvolvedMovieCast().forEach(cast ->
                result.add(cast.getForename())
            );
        }
        return result;
    }

    @Transactional  // Dirty Checking: kein save() nötig!
    public void changeMovieTitle(long id, String newMovieName) {
        if (!newMovieName.isEmpty()) {
            Movie tmp = this.movieRepository.findById(id).get();
            tmp.setTitle(newMovieName);
            // Hibernate erkennt Änderung und führt UPDATE automatisch aus
            // KEIN movieRepository.save(tmp) nötig!
        }
    }
}
Dirty Checking: Innerhalb einer Transaktion merkt sich Hibernate den Zustand aller geladenen Objekte. Beim Commit vergleicht es den aktuellen mit dem ursprünglichen Zustand und führt automatisch UPDATEs aus.
Standard-Transaktionen: Alle CRUD-Operationen des JpaRepository (save, findById, findAll, delete) laufen bereits automatisch als Transaktionen.

@Entity @Getter @Setter @NoArgsConstructor
public class Movie {
    @Id @GeneratedValue private long id;
    private String title;
    private double duration;
    private LocalDate premiereDate;
    private String description;
    private String trailerUrl;
    private String coverImgUrl;
    private double imdbRating;

    @ManyToOne(cascade = CascadeType.ALL)
    @JoinColumn(name = "director_id", nullable = true)
    private Director director;

    @ManyToMany(cascade = CascadeType.ALL)
    @JoinTable(name = "movie_moviecast",
        joinColumns =        { @JoinColumn(name = "movie_id",  referencedColumnName = "id") },
        inverseJoinColumns = { @JoinColumn(name = "cast_id",   referencedColumnName = "id") })
    private List involvedMovieCast;

    @OneToOne(mappedBy = "movie", cascade = CascadeType.ALL)
    private Soundtrack soundtrack;

    @OneToMany(mappedBy = "movie", cascade = CascadeType.ALL, fetch = FetchType.EAGER)
    private List awards;
}
